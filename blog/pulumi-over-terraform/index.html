Pulumi > Terraform

Before I joined TTI, my infrastructure-as-code (IaC) tool of choice was Terraform. Think of it as the brand-name Swiss Army knife for managing your infrastructure; in theory, it has all the tools you need, and everyone knows it. For exactly those reasons, I ignored my friend’s suggestions to explore Pulumi for quite some time. I planned to go full-speed ahead with Terraform at TTI. After all, any new DevOps engineer joining our team would likely be well-acquainted with Terraform.

However, wrestling with Terraform's HashiCorp Configuration Language (HCL) syntax can feel like learning a new language every time you use it after a break. It's one of those things that's easy to forget if you're not working with it regularly. On the other hand, I know Python and (some) JavaScript. So, in the spirit of experimenting, I decided to give Pulumi a shot. Lo and behold, I was pleasantly surprised.

I took up the task to spin up GitLab in EC2 using Pulumi and Javascript. The process was so seamless, so effortless, that I bid Terraform a swift goodbye. The ease and familiarity of JavaScript meant that not only could I cruise through configurations, but my developer colleagues could as well. And trust me, that's a real lifesaver when you're not around to troubleshoot.

To illustrate the difference, consider this: creating a new CNAME record in Cloudflare using JavaScript with Pulumi is as straightforward as:

if (stack.startsWith('qa-')) {
    const record = new cloudflare.Record(`${commitHash}-record`, {
        zoneId: "",
        name: `${commitHash}.qa.${domain}`,
        value: pagesProject.url,
        type: "CNAME",
        ttl: 1,
    });
}

Compare this to the same operation in HCL with Terraform:

resource "cloudflare_record" "record" {
  count        = var.stack_name != null && var.stack_name != "" && substr(var.stack_name, 0, 3) == "qa-" ? 1 : 0
  zone_id      = var.zone_id
  name         = "${var.commit_hash}.qa.${var.domain}"
  value        = var.pages_project_url
  type         = "CNAME"
  ttl          = 1
  proxied      = false
}

With Terraform, you're left forced to use a lengthy ternary operator comprised of substring functions and explicit index ranges. Not to mention that there's no easy way to handle conditional logic as Terraform doesn’t natively support the if-else construct, so ternary operations are as good as it gets.

But perhaps the most significant hurdles lie in Terraform's approach to environment management. While workspaces let you manage multiple environments, they all share the same configuration. That's akin to having a single blueprint for all the rooms in a building - it works until you need to customize a room. By leveraging a full-featured programming language like JavaScript, Pulumi enables me to use a single script to accommodate various environments.

These challenges that Terraform faces paint a vivid picture of why Pulumi, with its robust programming language support and more advanced state management, won my vote. As I architect some epic automation, dynamically creating production-like environments for QA, I cannot imagine grappling with these Terraform hurdles.

But that's a story for another day. So, stay tuned as we venture further into the dynamic infrastructure automation.
